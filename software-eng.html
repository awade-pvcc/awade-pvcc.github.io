Sofware Life Cycle: Development phase
  • Requirements analysis
    • The beginnning of the software life cycle and the purpose is to determine the services the system will provide, identify any conditions on those service, and to decide how users will interact with the system.
  • Design
    • This stage of the Software Life Cycle involves developing the internal structure of software system that can be converted into programs.
  • Implementation
    • This stage involves writing the programs, creating data files, and the development of databases.
  • Testing
    • This step in the Software Life Cycle is actually used in the three previous phases since the entire development process should be "tested" for accuracy. This three-step development process that has components such as "requirements analysis and confirmation," "design and validation," and "implementaion and testing."
modular implementation This method involves breaking down programs, large and small, with the objective of gaining understanding of the software. In the imperative paradigm, modules appear as functions, while in the object-oriented paradigm objects are used for the modules.
Unified Modeling Language (UML) A collection of tools that are used to show the relationship between the user and software. The use case diagram depicts the interactions with the use cases and actors. There are also class diagrams where a notational system is used to represent the structure and relationship of classes. Lastly, there is the interaction diagram that expresses events that occur during execution of the program.
use case The interactions within a system and its users.
use case diagram A display of the Unified Modeling Language as rectangles that show the interactions, or use cases, between the system and the users represented by ovals and users of the system, called actors, are showed as stick figures.
Data relationships:
  • one-to-one
    • Described as an association of one patient to one room and each room is associated to one patient.
  • one-to-many
    • Exemplified by an association between patient and physician where one physician is associated with many patients but each patient is associated with one physician
  • many-to-many
    • With the same type of example, this association would be each physician is associated with many patients and each patient is associated with many physicians.
design pattern A model developed as a resolution to a recurring problem the design of the software.
Software quality assurance (SQA) Groups that are tasked with overseeing and enforcing the quality control systems used by the organization.
Software Testing:
  • glass box testing
    • A technique based on the Pareto principle where the software tester uses their knowledge of the structure of the software to design the test.
  • black box testing
    • Testing that does not rely on the knowledge of the software's interior structure.
  • beta testing
    • A method that falls within the black box category where the preliminary verision of the software is given to a segment of the intended audience with the goal of learning how the software performs in real-life situations before the final version is put on the market.
  • alpha testing
    • Similar to beta testing but is performed at the developer's site.
Documentation:
  • user documentation
    • The explanation of the features of the software and description of how to use them.
  • system documentation
    • Describes the software's internal makeup so that the software can be maintained later in the life cycle.
  • technical documentation
    • Describes how a software system should be installed and maintained, which includes adjusting operating parameters, installing updates, and reporting problems back to the softwar's developer.
Intellectual Property Law:
  • software license
    • A legal agreement between the owner and user of a software product that allows the user permission to use the software without transferring ownership rights to the intellectual property.
  • software disclaimer
    • A form of protection for the developers of software where the limits of their liabiliy is explained.
  • best defense against software liability claims
    • Apply sound software engineering principles during the development, use a level of care compatible with the software's application, and to produce and maintain records that validate these points.
